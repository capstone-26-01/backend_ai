Issue Map Pi Harness Kimi Review Gate
=====================================

Purpose
-------

Phase 0 adds this runbook for the post-test AI review gate described in
ROADMAP.md. The gate is a release-quality review step after deterministic tests
pass. It is not part of the Django test runner, not application code, and not a
runtime issue-map dependency.

Provider
--------

- Harness: Pi harness review runner.
- Provider style: OpenAI-compatible.
- Model: opencode/kimi-k2.5.
- OpenCode Zen model id: kimi-k2.5.
- Endpoint: https://opencode.ai/zen/v1/chat/completions.
- Mode: one bounded review call, no tools, no agent loop.

Harness Evaluation Suite
------------------------

The reusable suite under `harness_eval/` checks whether model, tool, MCP, or
skill changes improve a real issue-map runner command before they are trusted
for production-like review gates. The command under test receives only a normal
issue job packet; hidden expected nodes/tool policy stay inside the deterministic
evaluator.

Offline checks:

- `python -m unittest harness_eval.tests`
- `python -m harness_eval.runner validate-samples`
- `python -m harness_eval.runner validate-matrix`
- `python -m harness_eval.runner render-job harness_eval/samples/origin_trace.json`
- `python -m harness_eval.runner run-harness harness_eval/samples/origin_trace.json -- <your-pi-command-and-args>`
- `python -m harness_eval.runner evaluate-transcript harness_eval/samples/origin_trace.json harness_eval/sample_transcripts/good_origin_trace.json`

Optional live OpenCode Zen smoke:

- set `OPENCODE_API_KEY` in the caller environment,
- set `RUN_OPENCODE_LIVE_TESTS=true`,
- run `python -m harness_eval.runner live-smoke --live --model kimi-k2.5 --write-result`.

The live smoke uses the raw model id `kimi-k2.5` against
`https://opencode.ai/zen/v1/chat/completions`. The OpenCode config id remains
`opencode/kimi-k2.5`. Keep these separate when debugging usage dashboards.
The smoke gate fails unless the provider response includes both `usage` and a
`response_id`; successful receipts are written under `temp/harness_eval/` with
timestamp/model/endpoint correlation fields. This proves the provider returned
usage metadata for the request, but it does not prove the web dashboard rendered
the request.

Production-like live Pi SUT eval:

- set `OPENCODE_API_KEY` in the caller environment,
- set `RUN_OPENCODE_LIVE_TESTS=true`,
- run `python -m harness_eval.runner run-harness harness_eval/samples/origin_trace.json -- python -m harness_eval.pi_runner --live --model kimi-k2.5`.

This keeps the deterministic evaluator outside the SUT. `pi_runner.py` invokes
Pi with only the bounded issue-map tools from
`harness_eval/pi/issue_map_extension.ts`, then extracts the transcript from the
terminating `finish_issue_map_transcript` tool result. If `live-smoke` passes
but this command fails, the provider is reachable and the failure is in the Pi
tool loop/output contract.

The finishing tool also clamps final nodes and paths to high-confidence
candidates derived from the job packet's own issue/artifact evidence. It must
not read hidden evaluator `expect` fields.

Credential Rules
----------------

- Store OpenCode Zen credentials only in the CI or reviewer environment.
- Do not put review credentials in Django settings, repo .env files, app code,
  fixtures, or test-runner configuration loaded by Django.
- Before enabling the gate in CI, record OpenCode Zen's retention/training
  policy. If that policy is unknown, treat every submission as permanently
  retained by an external provider.

Allowed Review Packet
---------------------

Send only a canonical review packet assembled from approved synthetic or
sanitized data:

- deterministic test summary,
- schema diff,
- representative synthetic /api/issues/ fixtures,
- representative synthetic /api/issues/related-nodes/ fixtures,
- graph invariant summary,
- ranking/eval summary,
- LLM fallback/sanitization summary,
- relevant PLAN.md and ROADMAP.md excerpts.

Forbidden Inputs
----------------

Never send:

- secrets,
- .env contents,
- raw credentials,
- production data,
- private repository contents,
- live GitHub responses,
- proprietary analyzed-repository code,
- real code_context excerpts from third-party repositories.

Only synthetic fixtures explicitly approved for third-party disclosure may be
included. Treat anything sent to OpenCode Zen as externally disclosed.

Run-Or-Waive Policy
-------------------

The Kimi review is advisory and must not mutate deterministic test results. It
is run when OpenCode Zen and the Pi harness are available. If the review is
unavailable for any reason, record an explicit human waiver in the phase merge
notes with the reason and timestamp.

Review Prompt Template
----------------------

Ask the reviewer to check the phase implementation against ROADMAP.md and
PLAN.md. The reviewer should focus on:

- phase-scope alignment,
- endpoint compatibility,
- no live network in automated tests,
- no issue-path caching,
- test fixture disclosure safety,
- schema/documentation coverage,
- workspace hygiene,
- whether the phase is safe to merge to dev.

The desired final line is either ACCEPT or a short list of material blockers.
