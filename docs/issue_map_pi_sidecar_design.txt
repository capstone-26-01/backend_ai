Runtime Pi Issue Harness Design
===============================

Status
------

Runtime design. The harness is optional and fail-closed, but it is no longer
only a deferred future note.

Boundary
--------

The Django backend remains the owner of:

- GitHub issue API calls,
- stored analysis artifact loading,
- deterministic evidence extraction,
- deterministic seed ranking,
- response schema compatibility,
- backend validation of all returned node IDs and paths.

The runtime harness owns:

- bounded repo investigation over the supplied artifact,
- tool-backed search/read/neighbor inspection,
- final issue-origin node selection,
- investigation path and confidence rationale.

Harness Input
-------------

`llm.issue_harness.build_issue_harness_job()` builds a bounded job packet:

- issue title/body and comment excerpts,
- extracted deterministic evidence,
- deterministic seed candidates as hints,
- graph nodes/edges from the stored artifact,
- bounded `file_contents` selected from the artifact,
- the list of available tools and safety rules.

The harness does not receive GitHub credentials, a repository clone path, raw
environment values, shell access, network access, unrestricted filesystem
access, or filesystem access outside the bounded job file.

Runtime Adapter
---------------

`llm.pi_issue_runner` invokes `@earendil-works/pi-coding-agent` with:

- JSON mode,
- no context files,
- no session,
- no built-in tools,
- a runtime extension at `llm/pi_issue_extension.ts`,
- allowlisted tools only.

The extension reads only the job file path provided through
`ISSUE_HARNESS_JOB_FILE`. Its tools expose bounded artifact data:

- `get_issue_context`
- `list_repo_files`
- `search_repo_symbols`
- `search_repo_text`
- `read_repo_file`
- `get_node`
- `get_neighbors`
- `finish_issue_map_transcript`

Validation And Fallback
-----------------------

`llm.issue_harness.run_issue_harness()` rejects harness output that:

- is not JSON,
- has no final fields,
- has no tool calls,
- exceeds the bounded tool-call budget,
- finishes without reading bounded issue context,
- finishes without listing bounded repository files,
- names origin nodes without repository search,
- names origin nodes without code or graph inspection.

`api.services.get_issue_map_response()` then validates returned node IDs against
the analysis graph. If the harness returns only missing nodes, fails, times out,
or violates the tool-use contract, the endpoint falls back to deterministic
ranking with a structured warning. No issue-path cache is introduced.

Frontend Contract
-----------------

Existing fields remain:

- `selected_node_ids`
- `candidates`
- `overview_graph`
- `focus_graph`
- `hypotheses`
- `investigation_path`
- `code_context`
- `confidence`

The additive `harness` field exposes bounded tool-call metadata for debugging.
The frontend may ignore it.

Testing
-------

Required deterministic tests:

- harness job packets contain bounded graph/file data,
- subprocess harness runner rejects no-tool final answers,
- subprocess harness runner rejects tool-call budget overruns,
- subprocess harness runner rejects final answers that never read issue context,
- subprocess harness runner rejects search-only final answers,
- default Pi runner fails closed without provider credentials,
- issue endpoint uses valid harness output to rebuild candidates/focus graph,
- issue endpoint falls back on invalid/missing harness output,
- full Django and harness-eval suites remain green.

Live Pi/OpenCode evaluation is separate from Django tests and must be explicitly
enabled before a merge/release gate.
